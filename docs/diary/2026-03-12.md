**Q: Go through entire project. Check all files. Any redundant code, or any issues which can cause performance issues as multiple people using the system?**
A: Full-project audit. 17 fixes across P0/P1/P2 categories:

**P0 — Security & Data Integrity**
1. `GET /api/auth/users` was returning raw password hashes from DB. Fixed: added `select: { id, name, email, avatar }` to Prisma query.
2. Any space member could edit/delete another member's memories. Fixed: ownership check before PUT/DELETE in `memories.ts` — non-owners/admins get 403.
3. `acceptInvite` had two sequential DB writes (upsert SpaceMember + delete PendingInvite) with no transaction. Fixed: wrapped in `prisma.$transaction([...])`.
4. Cloudinary upload had no file-type or size validation — anyone could upload arbitrary files. Fixed: `ALLOWED_TYPES` check + 10 MB cap in `cloudinary.ts` before sending to Cloudinary. Also moved `CLOUD_NAME` / `UPLOAD_PRESET` to `.env`.
5. Frontend `User` type had `password?: string` field — could leak password into state. Removed. Updated `login` store action signature to use a separate credentials type.

**P1 — Multi-user Correctness**
6. Reaction increment was a read-modify-write (findUnique → update) with race condition under concurrent clicks. Fixed: wrapped in `prisma.$transaction` with `SELECT ... FOR UPDATE` row lock.
7. `generateInviteCode` loaded ALL invite codes into memory then did JS `.includes()`. Fixed: `findFirst({ where: { inviteCode: candidate } })` — one targeted query per attempt.
8. `deleteSpace` in Zustand did optimistic removal with no rollback on API failure. Fixed: snapshot before remove, restore in catch + re-throw.
9. `addMemory`, `updateMemory`, `deleteMemory` were swallowing errors silently. Fixed: removed try/catch wrapping so callers receive the error.
10. Backend `/spaces/:id/invite` still restricted to owner/admin even though frontend allowed all members to invite. Fixed: changed check to `myMember.status === 'active'` to align with frontend.

**P2 — Performance & Cleanup**
11. `generateInviteCode` fixed (same as #7 above).
12. `setTimeout` calls missing cleanup — `SpaceSelector` password success timer and `Timeline` invite status timer both leaked on unmount. Fixed: added `useRef` + `clearTimeout` in cleanup `useEffect`.
13. Timeline `sortedMemories` was re-sorted on every render. Fixed: wrapped in `useMemo` keyed on `space?.id` + memory IDs.
14. `ParticleBackground` resize listener fired at full rate. Fixed: debounced with 100 ms delay + `clearTimeout` in cleanup.
15. `deleteSpace` route loaded full memory objects with `include: { substories: true }` just to collect photo URLs. Fixed: `select: { photos: true, substories: { select: { photos: true } } }`.
16. Backend had no `helmet` middleware (missing security headers) and no global error handler. Fixed: `app.use(helmet())` + express error handler at bottom of `index.ts`. Installed `helmet` package.
17. No rate limiting on `/invite` and `/react` endpoints. Fixed: added `actionLimiter` (30 req/min) for both routes.
18. `validatePassword` was duplicated in `SpaceSelector.tsx`, `LoginPage.tsx`, and `auth.ts` backend. Fixed: extracted to `src/utils/validation.ts`; updated SpaceSelector and LoginPage to import from there.

**Q: Owner of the group should give edit/view permissions to users who joined by invite. View → can only view memories and moments. Edit → can make changes as well.**
A: Added a `permission` field (`'view' | 'edit'`, default `'edit'`) to the `SpaceMember` model and DB. Full end-to-end implementation:

**Backend:**
- `schema.prisma`: added `permission String @default("edit")` to `SpaceMember`. Ran `prisma db push`.
- `db.ts`: `formatSpace` now includes `permission` in `membersList` output.
- `spaces.ts`: `PATCH /spaces/:id/members/:userId` now accepts `{ permission }` in addition to `{ role }`. Validates `'view' | 'edit'`, owner-only.
- `memories.ts`: Added `validateEditPermission()` helper — checks `permission === 'edit'` (owners always pass). Applied to all write routes: `POST /memories`, `PUT /memories/:id`, `DELETE /memories/:id`, `POST /substories`, `PUT /substories/:id`, `DELETE /substories/:id`. View-only members get 403.
- Also noted: removed invite code flow (join-by-code no longer used — invite by email + accept/reject from notifications only).

**Frontend:**
- `types/index.ts`: added `permission?: 'view' | 'edit'` to `SpaceMember`.
- `api.ts`: added `updateMemberPermission(spaceId, userId, permission)` → `PATCH` with `{ permission }`.
- `useStore.ts`: added `updateMemberPermission` action — calls API and updates local state immediately (no full refetch needed).
- `Timeline.tsx`: computed `myPermission` and `canEdit` from current user's member entry. Added `handleTogglePermission` + per-member pill toggle (teal `edit` / grey `view`) in `SpaceMembersPanel` — visible only to owner. Passed `canEdit` to `MemoryCard` and both `MemoryDetail` instances.
- `FloatingNav.tsx`: `onCreateClick` made optional — component returns `null` when not provided. Passing `undefined` when `!canEdit` hides the ➕ button entirely.
- `MemoryCard.tsx`: added `canEdit?: boolean` prop — wraps edit/delete hover buttons in `{canEdit && ...}`.
- `MemoryDetail.tsx`: added `canEdit?: boolean` prop — gates the ✏️ pencil toggle, the per-substory edit/delete buttons, and the "Add a moment" dashed button.
